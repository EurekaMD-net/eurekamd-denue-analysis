# v0.2.3 Ingest Plan — Decisions & Scope

**Fecha:** 2026-05-05
**Estado:** v0.2.3-A pendiente entrega operador (EDR ZIP).
**Predecesor:** [`fase-3-detalle.md`](./fase-3-detalle.md) (planning original 2026-05-03 con las 4 fuentes).

---

## Contexto

La Fase 3 original contemplaba 4 fuentes (Datatur + SINAIS + ENOE + ENIGH) como un solo bloque. Investigación de 2026-05-05 reveló que la granularidad real de cada fuente difiere mucho — solo 1 de las 4 hace join nativo a `cve_mun` con sample suficiente. Este documento congela esa decisión.

## Verdict por fuente

**Tres tiers de fit** (corrección operador 2026-05-05): no descartar fuentes que no joinen a `cve_mun`. Las que tienen llave a entidad/región/sector se cargan como **calibradores** — tablas paramétricas que condicionan / multiplican / contextualizan las filas municipales existentes, no nuevas filas en el join. Detalle en `feedback_calibrator_pattern` (memoria global).

| Fuente                              | Granularidad nativa                           | Cadencia        | Tier            | Decisión v0.2.3                                                                                                                                                                                                                            |
| ----------------------------------- | --------------------------------------------- | --------------- | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **EDR / SINAIS mortalidad** (INEGI) | Municipio (`ent_resid` + `mun_resid`)         | Anual, ~12m lag | **Direct join** | **A — ship next**. Único con granularidad municipal nativa y señal ortogonal a las 6 fuentes existentes (mortalidad complementa CLUES capacidad + CONEVAL pobreza + SESNSP violencia)                                                      |
| **Datatur** (SECTUR)                | Destino turístico (~50/70 mapean a `cve_mun`) | Mensual + anual | **Mixed**       | **B — defer**. ~50/70 destinos = direct-join (post-crosswalk ~1d). Los ~20 multi-municipio o sub-municipales = calibrador (occupancy% por zona turística como multiplier de demanda en municipios contenidos)                              |
| **ENIGH** (INEGI)                   | Entidad (n≈3,300 hogares/estado)              | Bienal          | **Calibrator**  | **C — load as state-level params**. No genera filas municipales (CONEVAL ya cubre pobreza @ cve_mun) pero **condiciona** la lectura: deciles de ingreso estatal calibran hasta dónde el `pobreza_pct` municipal refleja capacidad de gasto |
| **ENOE** (INEGI)                    | Entidad + 39 ciudades autorrepresentadas      | Trimestral      | **Calibrator**  | **C — load as state-level params**. Tasa de informalidad estatal multiplica counts de CE 2024 (más informalidad = más sub-reporte real); tasa de desocupación contextualiza `pea / pocupada` del Censo 2020                                |

**Resultado neto:** v0.2.3 = 1 direct-join (EDR) + 1 mixed deferred (Datatur, post-crosswalk) + 2 calibradores (ENOE + ENIGH como tablas paramétricas estatales).

**Patrón calibrador**: tablas separadas `calibrators_<source>_state` keyed por `entidad` (2-char). Endpoints existentes pueden `LEFT JOIN` por entidad para añadir campos `_calibrated` / `_adjusted` junto a las columnas raw. No tocan `cve_mun`-keyed views — son enrichment lateral.

## v0.2.3-A — EDR mortalidad

### Qué agrega

Mortalidad municipal: defunciones totales, tasa cruda por 1k habitantes, top causas (CIE-10), mortalidad infantil. Cierra el cuadrante "salud poblacional" (CLUES = capacidad instalada; EDR = outcomes).

### Source

- **Programa:** Estadísticas de Defunciones Registradas (EDR), INEGI
- **Anteriormente:** SINAIS mortalidad — INEGI absorbió la serie
- **Página:** https://www.inegi.org.mx/programas/edr/
- **Granularidad:** registro individual; campos `ent_resid` (entidad de residencia) + `mun_resid` (municipio de residencia 1-3 dígitos sin zero-pad). Construir `cve_mun = LPAD(ent_resid,2,'0') || LPAD(mun_resid,3,'0')`.
- **Variables clave:**
  - `causa_def` — código CIE-10 (5 chars, e.g. `I219`)
  - `edad_anios`, `sexo` — demografía básica
  - `ano_ocur`, `mes_ocur`, `ano_regis` — fechas
  - `lista_mex` — agrupador INEGI (215 categorías)

### Operator delivery (browser required)

INEGI bloquea descargas headless con un interstitial JS. Operador descarga vía navegador:

1. Abrir https://www.inegi.org.mx/programas/edr/#datos_abiertos
2. Bajar **"Conjunto de datos abiertos: 2023"** (ZIP, ~80-150 MB CSV interno)
3. Si 2023 aún no publicado para descarga abierta, usar 2022 (más reciente listado en página al 2026-05-05)
4. Soltar en `data/edr/conjunto_de_datos_defunciones_registradas_YYYY_csv.zip`
5. Avisar para correr el loader

Loader ya quedará escrito (ver siguiente sección) — solo espera el archivo.

### Schema target (Postgres)

```sql
CREATE TABLE inegi_edr_defunciones_raw (
  id BIGSERIAL PRIMARY KEY,
  ano_ocur INT NOT NULL,        -- 1985-2099 CHECK
  mes_ocur INT NOT NULL,        -- 1-12 CHECK
  ent_resid TEXT,               -- 2-char zero-pad
  mun_resid TEXT,               -- 3-char zero-pad
  cve_mun TEXT,                 -- ent_resid || mun_resid, NULL when no residencia
  sexo INT,                     -- 1=hombre 2=mujer 9=NE
  edad_anios INT,
  causa_def TEXT,               -- CIE-10 code
  lista_mex INT                 -- agrupador INEGI
);
CREATE INDEX inegi_edr_cve_mun_idx ON inegi_edr_defunciones_raw (cve_mun);
CREATE INDEX inegi_edr_ano_idx ON inegi_edr_defunciones_raw (ano_ocur);
```

### Mat-view target

```sql
CREATE MATERIALIZED VIEW mv_mortalidad_municipal_yearly AS
SELECT
  cve_mun,
  ano_ocur AS ano,
  COUNT(*)::bigint AS total_defunciones,
  COUNT(*) FILTER (WHERE edad_anios < 1)::bigint AS defunciones_menores_1ano,
  COUNT(*) FILTER (WHERE causa_def LIKE 'I%')::bigint AS causas_circulatorio,
  COUNT(*) FILTER (WHERE causa_def LIKE 'C%')::bigint AS causas_neoplasias,
  COUNT(*) FILTER (WHERE causa_def LIKE 'V%' OR causa_def LIKE 'W%' OR causa_def LIKE 'X%' OR causa_def LIKE 'Y%')::bigint AS causas_externas,
  COUNT(*) FILTER (WHERE causa_def LIKE 'E%')::bigint AS causas_endocrinas
FROM inegi_edr_defunciones_raw
WHERE cve_mun IS NOT NULL AND LENGTH(cve_mun) = 5
GROUP BY cve_mun, ano_ocur;

CREATE INDEX mv_mortalidad_yearly_cve_mun ON mv_mortalidad_municipal_yearly (cve_mun);
CREATE INDEX mv_mortalidad_yearly_ano ON mv_mortalidad_municipal_yearly (ano);
```

### Endpoints target (2 nuevos)

```
GET /analytics/mortality-summary?entidad=XX[&ano=YYYY]
  Per-municipio mortality profile: total + tasa per 1k pop (joining
  Censo 2020) + top causes + infant mortality. Mat-view-first w/ live
  fallback (mismo patrón que risk-summary).

GET /analytics/mortality-trend?cve_mun=NNNNN
  Annual mortality time series 2010-latest, single municipio.
```

### Loader scaffold (queue)

`scripts/load-edr.ts`:

- Reuse `RnidInput` discriminated-union pattern from `scripts/load-sesnsp.ts`
- Stream CSV via `head -1` for header + `iconv -f WINDOWS-1252 -t UTF-8 | tail -n +2 | tr -d '\r'` for body (encoding TBD; older años son W-1252, recientes UTF-8 — auto-detect via header BOM)
- COPY into raw table; derive `cve_mun` via `UPDATE` after COPY (NULL para muertes sin residencia válida)
- Refresh-matviews script: añadir `mv_mortalidad_municipal_yearly`

### Tests target

- ~10 nuevos tests en `analytics.test.ts` cubriendo ambos endpoints (success / 400 validation / 401 / mat-view fallback / malformed JSON)
- Loader unit tests para `cve_mun` derivation y header validation

## v0.2.3-C — ENOE + ENIGH como calibradores estatales

Corrección operador 2026-05-05: no descartar fuentes que no joinen a `cve_mun`. Las que tienen llave a entidad/región/sector se cargan como **calibradores** — tablas paramétricas estatales que condicionan / multiplican / contextualizan las filas municipales existentes. Patrón general en `feedback_calibrator_pattern` (memoria global).

### Schema target (calibradores)

```sql
CREATE TABLE calibrators_enoe_state (
  entidad TEXT PRIMARY KEY,           -- 2-char zero-pad
  trimestre TEXT NOT NULL,            -- e.g. '2025-Q4'
  tasa_desocupacion NUMERIC(5,2),
  tasa_informalidad_laboral NUMERIC(5,2),
  tasa_subocupacion NUMERIC(5,2),
  tasa_participacion NUMERIC(5,2),
  ingreso_promedio_mensual NUMERIC(12,2),
  poblacion_15_mas BIGINT,
  cargado_en TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE calibrators_enigh_state (
  entidad TEXT PRIMARY KEY,
  ano_levantamiento INT NOT NULL,     -- 2022, 2024, 2026...
  ingreso_corriente_promedio NUMERIC(12,2),
  gasto_corriente_promedio NUMERIC(12,2),
  decil_1_ingreso NUMERIC(12,2),      -- corte inferior
  decil_5_ingreso NUMERIC(12,2),      -- mediana
  decil_10_ingreso NUMERIC(12,2),     -- corte superior
  pct_hogares_inseguridad_alimentaria NUMERIC(5,2),
  pct_hogares_credito_formal NUMERIC(5,2),
  cargado_en TIMESTAMPTZ DEFAULT NOW()
);
```

### Cómo entran a los endpoints

Patrón: `LEFT JOIN calibrators_<source>_state ON entidad = LEFT(cve_mun, 2)`. Los handlers existentes (`/analytics/municipios`, `/analytics/risk-summary`, etc.) añaden columnas `_calibrated` / `_state_context` junto a las raw — no reemplazan.

**ENOE** — informalidad multiplica CE 2024:

```sql
-- En /analytics/municipios respuesta:
ce.personal_ocupado_total                              AS personal_ocupado,
ce.personal_ocupado_total *
  (1 + cal.tasa_informalidad_laboral / 100.0)::numeric AS personal_ocupado_calibrated,
cal.tasa_desocupacion                                  AS desocupacion_estatal_pct
```

Lectura: si la entidad tiene 30% informalidad, el real-economic-employment del municipio es ~30% mayor que lo reportado por CE 2024 (que solo cuenta formal).

**ENIGH** — deciles estatales contextualizan pobreza municipal:

```sql
-- En /analytics/municipios respuesta:
cv.pobreza_pct,
cal.decil_5_ingreso                                    AS mediana_ingreso_estatal,
cal.decil_1_ingreso                                    AS piso_ingreso_estatal,
-- "el municipio X tiene 40% pobreza, en un estado donde el decil-5 es 8,200/mes"
```

Lectura: 30% pobreza en CDMX (decil-5 ~$15k) vs 30% pobreza en Chiapas (decil-5 ~$5k) son fenómenos socioeconómicos distintos. ENIGH provee el ancla absoluta.

### Cadencia

- ENOE: 32 valores actualizados trimestralmente. Re-cargar cada Q (~6-week lag post-trimestre).
- ENIGH: 32 valores cada 2 años. Re-cargar al publicarse nueva onda (próxima ~Jul 2027 = ENIGH 2026).

### Endpoints nuevos (opcionales)

```
GET /analytics/state-calibrators?entidad=XX
  Combina filas de calibrators_enoe_state + calibrators_enigh_state
  para una entidad. Single-row response. Cache larga (max-age=3600).
```

Endpoints existentes pueden enriquecer respuestas vía LEFT JOIN sin nuevo endpoint — la decisión de exponer un endpoint dedicado depende de si el frontend quiere consumir calibradores aislados.

### Operator delivery

Misma situación de browser-gating que EDR. URLs:

- **ENOE**: https://www.inegi.org.mx/programas/enoe/15ymas/#datos_abiertos — un ZIP CSV por trimestre
- **ENIGH**: https://www.inegi.org.mx/programas/enigh/nc/2024/#datos_abiertos — bundle nacional + por entidad

Loaders procesan microdata, agregan a entidad (con `factor_expansion` para que la agregación sea representativa), insertan 32 filas en `calibrators_*_state`. ~½ día de trabajo cada uno una vez entregado el ZIP.
