# v0.2.15 + v0.2.16 — Estado-Grain Rollups for SICT and SEDATU

> Drafted 2026-05-10. Status: **plan only, no code shipped.**
> Closes deferred work from v0.2.13 (SICT) and v0.2.14 (SEDATU).

## TL;DR

Both v0.2.13 (SICT Datos Viales) and v0.2.14 (SEDATU SNIIV Financiamientos) shipped
muni-grain MVs only. The estado-grain rollups they explicitly deferred are now ready
to ship as two **independent, parallel** bundles:

- **v0.2.15** — `sict_traffic_by_estado` MV + extend `/analytics/entidad-detail`
  with nested `datos_viales` field. Coverage probably 31 of 32 estados (CDMX likely 0).
- **v0.2.16** — `sedatu_financing_by_estado` MV + extend `/analytics/entidad-detail`
  with nested `vivienda_financiamientos` field. Coverage ~32/32 (every estado has SOME
  2025 financing).

Each is ~1 day. They share no source code and can ship in either order. The plan
assumes **v0.2.15 first** (lighter — no separate base view needed).

CNBV (v0.2.12) already ships both grains via `cnbv_panorama_estatal`. SICT and SEDATU
mirror that pattern.

---

## Why estado-grain matters

`/analytics/entidad-detail` currently returns `population`, `religion`, `bienestar`,
`inclusion_financiera` (CNBV), and ~12 other categories. It does NOT yet return:

- TDPA / vehicle composition / federal-highway station counts at estado level
- Vivienda financing flows (acciones, monto, modality, organismo, vivienda tier mix) at estado level

For the analyzer's "compare two estados" workflow, both gaps matter. The muni-grain
MVs already exist; rolling them up to estado is the cheap finish.

## v0.2.15 — SICT Estado-Grain

### Critical design decision: aggregate from the **dedupe view**, not the muni MV

The muni-grain MV (`sict_traffic_by_municipio`) already has TDPA-weighted percentages
per muni. **Rolling those up to estado would be wrong** — averaging muni-level
percentages would weight a 5-station muni equal to a 100-station muni.

Instead, aggregate directly from `sict_estaciones_viales` (the post-spatial-join,
post-dedupe view, one row per station). This re-applies the TDPA-weighted formula
at estado level using station-level data.

### Schema (proposed)

```sql
CREATE MATERIALIZED VIEW sict_traffic_by_estado AS
SELECT
  SUBSTRING(sv.cve_mun, 1, 2)              AS cve_ent,
  COUNT(*)                                  AS station_count,
  SUM(sv.tdpa)::INTEGER                     AS tdpa_total,
  MAX(sv.tdpa)::INTEGER                     AS tdpa_max,
  ROUND(AVG(sv.tdpa))::INTEGER              AS tdpa_mean,
  -- TDPA-weighted vehicle composition (same formula as muni-grain, restated at estado)
  ROUND(SUM(sv.tdpa_motos)    * 100.0 / NULLIF(SUM(sv.tdpa), 0), 2) AS pct_motos,
  ROUND(SUM(sv.tdpa_autos)    * 100.0 / NULLIF(SUM(sv.tdpa), 0), 2) AS pct_autos,
  ROUND(SUM(sv.tdpa_buses)    * 100.0 / NULLIF(SUM(sv.tdpa), 0), 2) AS pct_buses,
  ROUND(SUM(sv.tdpa_camiones) * 100.0 / NULLIF(SUM(sv.tdpa), 0), 2) AS pct_camiones,
  ROUND(SUM(sv.tdpa_otros)    * 100.0 / NULLIF(SUM(sv.tdpa), 0), 2) AS pct_otros,
  COUNT(DISTINCT sv.clave)                  AS route_count,
  -- Top-3 routes by SUM(tdpa) within the estado
  (SELECT array_agg(clave ORDER BY clave_tdpa DESC)
     FROM (
       SELECT clave, SUM(tdpa) AS clave_tdpa
       FROM sict_estaciones_viales
       WHERE SUBSTRING(cve_mun, 1, 2) = SUBSTRING(sv.cve_mun, 1, 2)
       GROUP BY clave
       ORDER BY clave_tdpa DESC
       LIMIT 3
     ) t)                                    AS routes_top
FROM sict_estaciones_viales sv
WHERE sv.cve_mun IS NOT NULL
GROUP BY SUBSTRING(sv.cve_mun, 1, 2);
```

**Note**: `tdpa_motos`, `tdpa_autos` etc. need to be exposed on `sict_estaciones_viales`
view — currently the view exposes `pct_motos` (already-divided percentages). For estado
weighting we need station-level absolute volumes. Two options:

1. **Extend `sict_estaciones_viales`** to expose both raw (`tdpa_motos = tdpa * pct_motos / 100`)
   and the existing pct columns. Cleanest. ~5 lines added to view DDL.
2. Inline the multiplication in the rollup. Repeats logic but no DDL churn.

**Recommend option 1** — single source of truth for vehicle composition.

### Indexes

```sql
CREATE UNIQUE INDEX idx_sict_tbe_cve_ent ON sict_traffic_by_estado (cve_ent);
CREATE INDEX idx_sict_tbe_tdpa_total    ON sict_traffic_by_estado (tdpa_total DESC);
```

### Edge cases

- **CDMX (`cve_ent='09'`)** — likely 0 stations. Exclude from result via `WHERE`
  (32 max rows, possibly 31 if CDMX has zero federal highways). Surface as
  `datos_viales: null` in the handler (LEFT JOIN, mirror of muni-detail).
- **State-attribution semantics** — using `SUBSTRING(cve_mun, 1, 2)` derives estado from
  the _spatially-joined_ muni, NOT from the CSV-published `estado` column. This is
  intentional: it gives "traffic physically within this estado's borders" even if SICT
  published a station under a neighboring estado.

## v0.2.16 — SEDATU Estado-Grain

### Critical design decision: aggregate from a **new dedicated view**, not the existing muni-filtered view

The current `sedatu_financiamientos_2025` view filters out rows where `cve_mun` is
empty/non-numeric. That filter dropped **395 state-level "no distribuido por municipio"
rows** (11 national + 384 state-level catch-alls) at v0.2.14 ingest. For estado-grain
rollups we **want those rows** — they represent legitimate state-attributed financing
that wasn't municipality-broken-out.

**Proposed**: add a sibling view `sedatu_financiamientos_estado_grain_2025` that
filters only on `cve_ent` (not `cve_mun`), then aggregate from that.

```sql
CREATE OR REPLACE VIEW sedatu_financiamientos_estado_grain_2025 AS
SELECT
  TRIM(cve_ent)                                    AS cve_ent,
  ano::INTEGER                                     AS ano,
  organismo::INTEGER                               AS organismo,
  modalidad::INTEGER                               AS modalidad,
  destino::INTEGER                                 AS destino,
  vivienda_valor                                   AS vivienda_valor,
  sexo::CHAR(1)                                    AS sexo,
  tipo_credito::CHAR(1)                            AS tipo_credito,
  acciones::INTEGER                                AS acciones,
  NULLIF(monto_pesos, '')::NUMERIC(20,2)           AS monto_pesos
FROM sedatu_financiamientos_raw_2025
WHERE NULLIF(TRIM(cve_ent), '') IS NOT NULL
  AND TRIM(cve_ent) ~ '^(0[1-9]|[12][0-9]|3[0-2])$';
  -- NO cve_mun filter — keeps state-level catch-alls
```

This adds ~395 rows back into the estado-grain aggregate that the muni-grain view
correctly excluded. The estado-MV's `acciones_total` will be **larger** than
`SUM(acciones_total)` over the muni-MV — by exactly the catch-all volume. Document
this in handler comment.

### MV schema (proposed)

```sql
CREATE MATERIALIZED VIEW sedatu_financing_by_estado AS
WITH organismo_totals AS (
  SELECT cve_ent, organismo, SUM(acciones) AS acc
  FROM sedatu_financiamientos_estado_grain_2025
  GROUP BY cve_ent, organismo
),
top_organismo AS (
  SELECT DISTINCT ON (cve_ent)
    cve_ent, organismo AS top_organismo_code, acc AS top_acc
  FROM organismo_totals
  ORDER BY cve_ent, acc DESC
)
SELECT
  f.cve_ent,
  MIN(f.ano)::INTEGER::TEXT                                      AS periodo,
  SUM(f.acciones)::INTEGER                                       AS acciones_total,
  SUM(f.monto_pesos)::NUMERIC(20,2)                              AS monto_total,
  ROUND(AVG(f.monto_pesos), 2)                                   AS monto_per_accion_avg,
  to_.top_organismo_code,
  org.nombre                                                     AS top_organismo_nombre,
  ROUND(to_.top_acc * 100.0 / NULLIF(SUM(f.acciones), 0), 2)     AS top_organismo_share,
  -- Modality breakdown (COALESCE-guarded for zero-rows, mirror v0.2.14 fix)
  COALESCE(SUM(f.acciones) FILTER (WHERE f.modalidad = 1), 0)
    * 100.0 / NULLIF(SUM(f.acciones), 0)                         AS pct_vivienda_nueva,
  COALESCE(SUM(f.acciones) FILTER (WHERE f.modalidad = 2), 0)
    * 100.0 / NULLIF(SUM(f.acciones), 0)                         AS pct_mejoramientos,
  COALESCE(SUM(f.acciones) FILTER (WHERE f.modalidad = 3), 0)
    * 100.0 / NULLIF(SUM(f.acciones), 0)                         AS pct_vivienda_usada,
  COALESCE(SUM(f.acciones) FILTER (WHERE f.modalidad = 4), 0)
    * 100.0 / NULLIF(SUM(f.acciones), 0)                         AS pct_otros,
  -- Demographics
  COALESCE(SUM(f.acciones) FILTER (WHERE f.sexo = 'M'), 0)
    * 100.0 / NULLIF(SUM(f.acciones) FILTER (WHERE f.sexo IN ('M','H')), 0) AS pct_femenino,
  COALESCE(SUM(f.acciones) FILTER (WHERE f.tipo_credito = 'I'), 0)
    * 100.0 / NULLIF(SUM(f.acciones) FILTER (WHERE f.tipo_credito IN ('I','C')), 0) AS pct_credito_individual,
  -- Vivienda tier (subtree-NULL when no rows have valor — same logic as muni-grain)
  COALESCE(SUM(f.acciones) FILTER (WHERE f.vivienda_valor = 'Económica'), 0)
    * 100.0 / NULLIF(SUM(f.acciones) FILTER (WHERE f.vivienda_valor IS NOT NULL), 0) AS pct_economica,
  -- ... 5 more tiers (Popular / Tradicional / Media / Residencial / Residencial Plus)
FROM sedatu_financiamientos_estado_grain_2025 f
JOIN top_organismo to_ USING (cve_ent)
LEFT JOIN sedatu_organismos org ON org.codigo = to_.top_organismo_code
GROUP BY f.cve_ent, to_.top_organismo_code, to_.top_acc, org.nombre;
```

### Indexes

```sql
CREATE UNIQUE INDEX idx_sedatu_fin_est_cve_ent     ON sedatu_financing_by_estado (cve_ent);
CREATE INDEX        idx_sedatu_fin_est_monto_total ON sedatu_financing_by_estado (monto_total DESC);
```

### Edge cases

- **National catch-all rows (`cve_ent='00'` or NULL)** — already filtered by view regex.
  Estado MV stays at 32 rows max.
- **Estado with 0 financing rows** — extremely unlikely (every estado had financing
  activity in 2025). LEFT JOIN handles it gracefully if it happens.
- **Discrepancy SUM(muni) ≠ estado** — by ~395 rows nationally. Document in
  handler comment so analyzer/operators don't flag this as a bug.
- **Vivienda tier nulls** — same handler logic as muni-grain (subtree-NULL when all
  6 tier %s are null). Reuse `viviendaFinanciamientosFromRow` marshaller.

## Cross-cutting items

### 1. Recurring regression: MV-refresh-omission caught in v0.2.13 AND v0.2.14 audits

Both ingests forgot to add their new MV to `refresh-matviews.sh`. Round-2 audit
caught both as C1 issues. **Third time would be embarrassing**. Two prevention paths:

#### Option A — Loader-level test assertion (recommended)

In each loader's `*.test.ts`, add:

```typescript
it("refresh-matviews.sh references the MV this loader creates", () => {
  const script = fs.readFileSync(
    path.join(__dirname, "refresh-matviews.sh"),
    "utf-8",
  );
  expect(script).toMatch(
    /REFRESH MATERIALIZED VIEW [\w.]*sict_traffic_by_estado/,
  );
});
```

Cheap, runs in unit-test scope, fails fast at PR-time if the operator forgets.

#### Option B — Pre-commit hook scanning all loaders

A `scripts/audit-mv-refresh.sh` that:

1. Greps `scripts/load-*.ts` for `CREATE MATERIALIZED VIEW <name>`
2. Confirms each `<name>` appears in `scripts/refresh-matviews.sh`
3. Exits non-zero if mismatch

Heavier but catches drift across the whole tree. **Defer to v0.2.17** if Option A
proves insufficient.

**Recommend shipping Option A in BOTH v0.2.15 and v0.2.16 as part of the bundle.**

### 2. Audit cadence

Continue 2-round `qa-auditor` per `feedback_round2_audit_pattern.md`:

- Round 1 — fresh-eyes review of the bundle as committed
- Round 2 — re-audit AFTER fixes, looking for contract-level drift introduced by the fix itself
- Round 3 only if round 2 catches a Critical that has further-reaching follow-ups

Both bundles are small (~150-200 lines) so 2 rounds should suffice.

### 3. Documentation

Each bundle:

- Adds a new entry in `docs/v0.2-status.md`
- Updates the project memory file (`project_denue_data_analysis.md`) with description bump
- Considers a feedback memory pin if a NEW gotcha is uncovered (none expected — both
  patterns are repetitions of what v0.2.13/v0.2.14 already pinned)

**v0.2.15 first, then v0.2.16 same-day if v0.2.15 audit closes clean.**

Reasons:

- v0.2.15 is lighter (~150 LOC, no separate base view)
- v0.2.15 verifies the SICT TDPA-weighted formula at a second grain (sanity check on muni-grain)

Estimated end-to-end (both bundles): ~6-8h dev + 2 audit rounds each ≈ 1 working day.
