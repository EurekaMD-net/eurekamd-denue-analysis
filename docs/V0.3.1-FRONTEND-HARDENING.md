### E2 — AGEB-grain fields reachable (`/analytics/locust-ageb` + selector)

Returns `{cve_mun, municipio, agebs:[{cvegeo, cve_ageb, pobtot_ageb, grado_rezago_ageb}]}`. - **`EndpointDef.path` signature extended** from `(entidad) => …` to `(ctx: EndpointPathCtx) => …` where `EndpointPathCtx = {entidad, cveMun}`. All 7 existing endpoints destructure `entidad` and ignore `cveMun`. New `needsCveMun` flag on `EndpointDef`.
- **New X-identifier field** `censo.cve_ageb` (`categorical_nominal`, grain `ageb`) — the two AGEB data fields are Y/Z-shaped; a chart needs an AGEB identifier to anchor on. All 3 AGEB fields wired to `locust-ageb`.
- **`MunicipioSelect`** component in LocustMode: a municipio dropdown (fed by `/analytics/municipios?entidad=`) shown when X's active endpoint `needsCveMun`. `cveMun` is local LocustMode state, cleared whenever the store `entidad` changes. Two sequential pre-fetch hints: pick entidad → pick municipio.

### E1 — `sesnsp.ano` deferred (design fork, written reason)

The punch-list line read "3 fields need `/risk-trend` + ageb selector" — but research found **`/analytics/risk-trend` already exists**. It returns a _single-municipio monthly time-series envelope_ (`{cve_mun, municipio, series:[{ano, mes, …}]}`), which is structurally incompatible with Locust's multi-row-per-grain X/Y/Z model. Wiring `sesnsp.ano` as a Locust field would either misrepresent the data or require a real new capability — exactly the "UI claims something it doesn't deliver" anti-pattern the honesty pass just removed.

**Deferred with reason.** To make `sesnsp.ano` reachable, a future cycle must pick one:

1. A Locust **temporal sub-mode** — when X is `temporal`, the chart scopes to one municipio (reusing `MunicipioSelect`) and fetches `/risk-trend` directly, bypassing the X/Y/Z resolver.
2. A new **`/analytics/risk-trend-estado`** endpoint that aggregates crime yearly by entidad, returning flat `(entidad_or_muni, ano, count)` rows that DO fit the X/Y/Z model.

Neither is hardening; both are features. `sesnsp.ano` keeps an empty `endpoints` map and a corrected field description (the old one wrongly said `/risk-trend` was "no servido aún").

**Audit**: single qa-auditor pass — verdict **PASS WITH WARNINGS** (0 critical, 2 warnings), both adjudicated as non-fixes after verification:

- **W1** (`useLocustDataset` query key omits `zField.id`) — **verified false positive.** The auditor's premise was "the path changes when Z changes" — but `path` is `endpoint.path({entidad, cveMun})` and takes no Z. Z's only fetch-relevant effect is on endpoint resolution (`getActiveEndpoint(x, y, z)`), and the resolved `endpointId` IS in the query key. If Z flips the endpoint → key changes → refetch. If Z doesn't flip the endpoint → no refetch needed (the payload carries all columns; Z's column-selection is client-side in `extractRows`). Adding `zField.id` would be pure redundancy. This is also the pre-existing key shape, unchanged by E2 except for the `cveMun` addition.
- **W2** (`MunicipioSelect`: `cveMun` not reconciled if it falls out of the muni list) — **deferred, cosmetic.** `cveMun` is only ever set by picking from this very list (same entidad) and is cleared on entidad change — so it's always either null or a valid same-entidad municipio. The only path to "set but not in list" is a `municipios` query error on background refetch, where the chart still renders valid same-entidad data and only the dropdown looks empty. Not a data-correctness bug; a reconciliation `useEffect` is moderate effort for a cosmetic edge. Add it if a user reports the empty-dropdown confusion.
